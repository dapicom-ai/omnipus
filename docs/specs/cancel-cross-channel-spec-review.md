# Cross-Channel `/cancel` — Spec Review (Adversarial)

**Spec under review:** `docs/specs/cancel-cross-channel-spec.md`
**Mode detected:** `plan-spec` (full BDD scenarios, FR-IDs, SC-IDs, traceability matrix, ambiguity log)
**Review date:** 2026-05-14
**Reviewer constraint:** Read-only — findings only, no spec edits.

---

## 0. Executive Summary

**Verdict: BLOCK.**

This spec reads cleanly and the decision log is unusually thorough, but it
rests on three load-bearing assumptions that the codebase does not currently
support. Each of those is a CRITICAL finding on its own:

1. **Sub-turns will be cancelled via `transcriptSessionID` cascade — but sub-turns do not have `transcriptSessionID` populated.** `subturn.go` builds `processOptions` without setting `TranscriptSessionID`, so every child `turnState` is registered into `activeTurnStates` with an empty string in that field. The cascade walk the spec describes (`ts.transcriptSessionID == sessionID`) will match **zero** sub-turns. The headline US-1.2 / US-4.1 / SC-6 outcomes ("parent + N sub-turns exit within 3 seconds") are unreachable without first wiring this field, and the spec does not call that out.
2. **Handoff cascade is described as cancelling a concurrently-running post-handoff turn — but handoff is a *next-message* routing override, not a concurrent agent.** No "Ray's turn" runs while Mia's turn is in flight. US-4.2 and BDD scenario "Handoff target is included in cascade" describe a runtime topology the agent loop does not produce.
3. **`InterruptHard()` currently has no session parameter and walks `getAnyActiveTurnState()`.** The spec's two-stage escalation (FR-11) says "hard cancel on every still-active matching turnState" but the existing primitive can't target a session, let alone cascade. Reusing it as-named is impossible; either rename and resignature, or document the new API surface. The spec lists `InterruptHard` as "CALLED — Existing primitive — reused" with no signature change — that's wrong.

There are also four MAJOR findings (transcript-vs-context store conflation, slash-menu unreachable during streaming, FR-8 vs FR-7 contradiction on pending-approval lifecycle, audit-attempt-entry contradiction with no-active-turn no-op), six MINOR findings, and a handful of observations. Detail follows.

Counts: **CRITICAL 4 · MAJOR 4 · MINOR 7 · OBSERVATION 5 · Total 20**.

## 3. Test Coverage Gap Analysis (plan-spec mode)

Strengths:
- Per-tier text-parse table (T11-D1) is exhaustive.
- Mock-clock testing for two-stage timer (T6, T7) is the right approach.
- Cross-channel cascade matrix (T1-D1) covers the orthogonal axes.

Gaps:

- **No test asserts sub-turn `transcriptSessionID` inheritance.** Closely related to F-01 — a regression test that fails on current `main` (because the field is empty) is the single most important gate test for this feature. Add as T0 or T1a.
- **No test for handoff-cascade behaviour after the architectural correction in F-02.** Either the test must be removed (option a) or replaced with one that drives the new spawn-style handoff topology (option b).
- **No negative test for the `InterruptHard` cascade behaviour** (F-03). T6 only asserts "InterruptHard was called" but does not specify *on which turnStates*. A test that registers two turnStates with the same session_id and asserts both received hard-abort is missing.
- **No test for the partial-content-on-context-store vs flag-on-transcript-store separation** (F-04). T9 asserts the transcript file has the truncated flag but does not verify `context.jsonl` was *not* mutated with the flag (the LLM next-turn correctness invariant).
- **No test for FR-20 conflict resolution** (F-07). Once resolved, a test should assert `turn_cancel_attempt{was_fired: false}` exists when cancel is issued against an idle session.
- **No abuse-vector test for FR-25 / Q5** (F-08). At minimum a test where User A starts a turn, User B issues `/cancel`, assert audit log captures B as canceller. Currently T13 mentions canceller identity in passing but does not drive the multi-user scenario directly.
- **No CI test for the slash-menu-during-streaming invariant** (F-05). If the spec sticks with US-1.3, an integration test must verify the slash menu opens during streaming when the user types `/`.
- **CLI test T16 spawns a subprocess** — `omnipus agent` interactive mode. This requires a TTY in CI. Standard GitHub Actions runners do not allocate a TTY by default. Note this limitation; consider testing the raw-stdin handler in isolation via T17 (Go unit test) plus a manual checklist for the subprocess case.

## 5. Unasked Questions

The spec should have answered these but does not:

1. **How is sub-turn `transcriptSessionID` actually wired?** The cascade fundamentally depends on it. See F-01.
2. **What happens to in-flight LLM streaming HTTP requests on cancel?** Spec mentions "context cancellation aborts in-flight HTTP request" in §7 but does not specify the provider-cancel path (`turnState.setProviderCancel`, `turn.go:292`). Without it, a graceful cancel must wait for the streaming response to complete naturally — that can be 5-20 seconds with OpenRouter, blowing the 3s graceful window. Does the graceful cancel call `providerCancel()` or wait for the natural stream end?
3. **What does the SPA UI show during the 3s graceful window for a stuck cancel?** Spec says "Stop button morphs to Stopping..." (FR-21) but does not say what happens at the 3s mark when the button is still showing "Stopping..." Is it then "Force-stopping..."? Stays "Stopping..."? Re-enabled?
4. **What if a sub-turn is in the middle of its own LLM stream when parent cancel fires?** Sub-turns have their own `providerCancel`. Does cascade hit each child's `providerCancel`? Spec doesn't say.
5. **Cost accounting suppression (FR-17) — when does it begin?** From the detach moment (5s after hard) — but what about cost incurred between cancel-issued and detach? Spec implies that cost counts, but doesn't say. Operators billing per-token care about this boundary.
6. **Replay rendering of a *previously cancelled* session (FR-16) — what is displayed?** Spec says the `turn_cancelled` entry is filtered from chat bubbles. But the `(interrupted)` suffix on the partial assistant message must survive replay. Does the SPA render the `(interrupted)` label from the `truncated: true` flag on the transcript entry, or from a separate UI marker? T10 only verifies absence of bubble, not presence of suffix.
7. **What is the user-facing chat-channel message when cancel races with completion (EC-2, last-write-wins=natural)?** Spec says "no `turn_cancelled` entry is written" — does the bot still send "Cancelled by @user"? Or do users get no acknowledgement at all? Either answer is fine, but the spec doesn't pick one.
8. **Does cancel propagate to MCP tool calls in flight?** FR-8 carves out "MCP servers" as not killed, but does the *in-flight RPC* to an MCP server get a cancel signal (context cancel), or does the agent loop just stop reading the response while the MCP server keeps computing? Operators of expensive MCP servers (paid model providers) care.
9. **Idempotency of audit emits under simultaneous cross-channel cancel (EC-4).** Spec says "one fired audit entry, two attempt entries." How is the deduplication implemented? Per-session cancel-mutex with an `cancelFired` flag? Spec EC-4 narrates the outcome but FR section does not specify the mechanism (related to F-13).
10. **MaixCam config migration — do we drop the field silently from the on-disk config.json or rewrite it on next save?** FR-34 says "log a warning, ignore the section." Does the next config write strip the section? If so, downgrade compatibility is broken. If not, the warning fires on every boot forever.

## Verdict

**BLOCK.** Four CRITICAL findings render core acceptance scenarios untestable (F-01, F-02, F-03) or implementing them on the wrong file (F-04). Four MAJOR findings need spec-level resolution before implementation. The remainder are quality-grade improvements.

To address these findings, run:

```
/plan-spec --revise docs/specs/cancel-cross-channel-spec.md docs/specs/cancel-cross-channel-spec-review.md
```

Suggested response priority for the spec author:
1. **F-01, F-02, F-03** — fundamental architecture corrections. Each may add 1-3 FRs and shift the symbols table.
2. **F-04** — transcript-vs-context store separation; this rewrites §8 Flow D and likely FR-14/FR-15.
3. **F-05, F-06, F-07** — resolve contradictions / unreachable UI paths.
4. **F-08** — explicitly decide whether channel-scoped cancel auth is acceptable for v0.1; if yes, write the abuse-detection FR (F-08 fix b).
5. Remaining MINOR/OBSERVATION items can be addressed in the same revision or deferred with rationale.

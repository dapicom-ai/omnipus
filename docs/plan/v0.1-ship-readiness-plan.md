1. **`ws-reconnect.spec.ts` is a real source bug.** `src/lib/ws.ts:496` attaches `visibilitychange` to `window`. The Web Platform spec only fires this event on `document`. The test (and any real browser) dispatches it on `document`, so the listener never fires. Fix is one line: `document.addEventListener('visibilitychange', this._onVisibilityChange)` and matching `removeEventListener`. The `online` listener (`window.addEventListener('online', вҖҰ)`) is correct as-is вҖ” `online`/`offline` ARE `window` events.
2. **`agents.spec.ts:89` and `skills.spec.ts:44` are test bugs, not source bugs.** `CreateAgentModal.tsx:124` already feeds `isApiError(err) ? err.userMessage : вҖҰ` into the toast. `userMessage` for status 409 is `"This conflicts with the current state. Please refresh and try again."` (`src/lib/api-error.ts:50`). The test asserts `text=409`, which never appears. Fix: change the locator to `text=conflicts with the current state`.
3. **`retention.spec.ts:194` hits the wrong URL.** Test POSTs `/api/v1/admin/retention-sweep`; production endpoint is `/api/v1/security/retention/sweep` (`pkg/gateway/rest.go:2149`, handler at `pkg/gateway/rest_retention.go:163`). Fix: update test URL.
4. **`ExecApprovalBlock.tsx:108` already has `data-testid="approval-modal"`** and `:195` already has `data-testid="always-allow-toggle"`. Allowlist entries #106 and #108 are NOT blocked on adding the testid вҖ” they are blocked on the test's ability to deterministically trigger a `policy=ask` tool call. The fix is on the test side (set up an agent with `experimental.workspace_shell_enabled=true` + a workspace policy of `ask` for `workspace.shell`, then issue a deterministic prompt) plus persistence wiring for "Always Allow" toggle (verify it actually mutates the per-agent policy).
